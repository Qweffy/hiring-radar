# Prompt Injection Defense + LLM Output Validation — Best Practices
> Researched 2026-06-10. Sources verified against current official docs.

## TL;DR
- Every HN posting is hostile input. Spotlight it (randomized delimiters/datamarking), pass it only as data — never let it carry instructions ([MSRC](https://www.microsoft.com/en-us/msrc/blog/2025/07/how-microsoft-defends-against-indirect-prompt-injection-attacks)).
- Zod-validate (zod 4, `z.strictObject` + `z.toJSONSchema`) every model output server-side before any DB write or tool execution. Groq `strict: true` constrained decoding is a layer, not a substitute.
- Agent tools: static allow-list, read-only, zod-validated args. Never combine untrusted content + private data access + external communication in one context (the [lethal trifecta](https://simonwillison.net/2025/Jun/16/the-lethal-trifecta/)).
- Rate-limit every route that triggers a model call with `@upstash/ratelimit` sliding windows, per identity, checked *before* the model call.
- Keep a checked-in seeded attack corpus; run promptfoo as a CI regression suite with canary-token assertions. Every bypass found becomes a permanent test.
- Assume some injections succeed — [OWASP LLM01:2025](https://genai.owasp.org/llmrisk/llm01-prompt-injection/) states there is no foolproof prevention. Design for blast-radius containment.

## Practices
- **Spotlight untrusted content.** Wrap each HN posting in per-request randomized delimiters (e.g. `<data-{nanoid}>…</data-{nanoid}>`) and tell the model in the system prompt that everything inside is data, never instructions. Strip the delimiter sequence from the content first. Microsoft's spotlighting (delimiting/datamarking/encoding) cut indirect injection ASR from >50% to <2% on GPT-family models ([paper](https://arxiv.org/abs/2403.14720), [MSRC](https://www.microsoft.com/en-us/msrc/blog/2025/07/how-microsoft-defends-against-indirect-prompt-injection-attacks)). Datamarking (interleave `^` between words) is a stronger variant for the parser, where prose fidelity in the prompt doesn't matter.
- **Enforce instruction hierarchy by message role.** All instructions live in the `system` message; postings and tool results go in `user`/`tool` messages as spotlighted data. Models are trained to prioritize system > user > tool ([OpenAI](https://openai.com/index/prompt-injections/)) — but this is probabilistic, so keep the other layers.
- **Harden the parser as a pure extraction task.** One posting per call, no tools, no conversation history, fixed schema. Use Groq structured outputs with `strict: true` (token-level constrained decoding, guarantees schema shape) — currently only on `openai/gpt-oss-20b`/`-120b`; other models are best-effort ([Groq docs](https://console.groq.com/docs/structured-outputs)). Build the schema once: `z.toJSONSchema(postingSchema)` (native in zod 4 — do not install `zod-to-json-schema`).
- **Always `zod.parse` before persistence — constrained decoding is not validation.** `strict: true` guarantees shape, not content. Server-side: `z.strictObject` everywhere, `.max()` on every string, enums for closed sets, URLs restricted to `http(s)` and length-capped. On failure: log, quarantine the raw posting (status `parse_failed`), continue the batch. Never loosen the schema or echo model errors back for a free-form "fix it" retry. [OWASP LLM01 mitigation 2](https://genai.owasp.org/llmrisk/llm01-prompt-injection/): "define and validate expected output formats with deterministic code".
- **Treat validated output as still-untrusted text (LLM05 improper output handling).** A schema-valid `description` field still contains attacker text. Render with React text nodes or a markdown renderer with raw HTML disabled; block markdown images and auto-linked URLs from model/posting text — image URLs are the classic exfiltration beacon Microsoft blocks deterministically ([LLM05](https://genai.owasp.org/llmrisk/llm052025-improper-output-handling/)).
- **Agent tool design: allow-list + least privilege (LLM06 excessive agency).** Hand-rolled agent = you own the loop: dispatch only over a static map of tool names; zod-validate tool args before execution; tools are parameterized read-only DB queries (`search_jobs`, `get_posting`). No fetch-arbitrary-URL, no write, no email tool in any context that has seen posting text — that completes the lethal trifecta. Cap agent iterations (e.g. 6) and total tool calls per request. Tool results that contain posting text get re-spotlighted before re-entering the prompt.
- **Direct injection: user input is also untrusted.** Search queries and chat messages go through the same delimiting, length caps (~2k chars), and never get interpolated into the system prompt. An optional cheap pre-filter (regex/classifier for known patterns) is telemetry, not a gate.
- **MCP server = another untrusted client surface.** Reuse the exact same tool handlers and zod schemas; authenticate callers and apply the same rate limits. Agentic risks (tool misuse, goal hijacking) are now formalized in the [OWASP Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/).
- **Rate limit AI endpoints with Upstash.** `@upstash/ratelimit` + `@upstash/redis`, `Ratelimit.slidingWindow(10, "60 s")` keyed by `userId ?? ip` for search/agent routes, stricter (e.g. `3/60s`) for the agent; check before any model call and return 429 with `Retry-After` ([Upstash](https://upstash.com/blog/nextjs-ratelimiting)). Add a fixed-window daily global budget key to cap total spend — an agent loop multiplies model calls per HTTP request, so count model invocations, not just requests.
- **Build the injection regression suite around a seeded corpus.** Check in `tests/injection/corpus/*.json`: realistic HN-style postings each embedding one attack class — instruction override ("ignore previous…", roleplay), fake system/tool messages, tool-invocation requests, markdown-image/URL exfiltration with a canary token, base64/ROT13 payloads, zero-width unicode, delimiter-escape attempts, second-order payloads aimed at the agent reading stored data. Run [promptfoo](https://www.promptfoo.dev/docs/red-team/) (`npx promptfoo eval --no-cache` in CI) against the real parse function and agent endpoint via a JS/HTTP provider. Assert per test: output passes the zod schema; no system-prompt canary string leaks; no unexpected tool calls; exfiltration canary URL never appears in output or tool args. Also run `promptfoo redteam` periodically with the `prompt-injection`/`indirect-prompt-injection` plugins to discover new bypasses; promote every finding into the static corpus so it can never regress.

## Pitfalls
- **Regex blocklists ("ignore previous instructions") as the primary defense.** Trivially bypassed via paraphrase, translation, encodings. Fine as logging/telemetry only.
- **Static delimiters (```` ``` ````, `<data>`) copied from 2023 tutorials.** The attacker just includes the closing delimiter in the posting. Randomize per request and strip the sequence from content.
- **Equating JSON mode with validation.** `response_format: json_object` guarantees syntax only; Groq `strict: false` "may produce valid JSON that does not match your schema" ([Groq docs](https://console.groq.com/docs/structured-outputs)). The zod parse before the Drizzle insert is non-negotiable.
- **LLM-as-injection-detector as the sole gate.** The classifier prompt is itself injectable. Acceptable as one probabilistic layer (cf. Microsoft Prompt Shields), never the boundary.
- **Believing spotlighting "solves" injection.** It reduces ASR; OWASP explicitly says prevention is not foolproof. The deterministic layers (schema validation, tool allow-lists, output handling) are what bound the damage.
- **zod v3 idioms.** `.passthrough()`/`.strict()`/`.strip()` are deprecated in zod 4 — use `z.strictObject()`/`z.looseObject()`; `zod-to-json-schema` is obsolete ([zod v4 changelog](https://zod.dev/v4/changelog)).
- **Rate limiting only HTTP requests at the edge.** One agent request can fan out into N model calls; budget model invocations too.
- **Stored (second-order) injection.** Raw posting text persisted today and fed to a different LLM stage later (re-embedding, agent summaries) without re-spotlighting reactivates the payload. Spotlight at every model boundary, not just ingestion.
- **Retry loops that feed validation errors + raw output back to the model unbounded.** Cap at 1–2 retries with the same strict prompt, then quarantine.

## Version notes
- **OWASP**: LLM Top 10 **v2025** is current ([genai.owasp.org](https://genai.owasp.org/llm-top-10/)); relevant entries: LLM01 prompt injection, LLM02 sensitive info disclosure, LLM05 improper output handling, LLM06 excessive agency. New since Dec 2025: **OWASP Top 10 for Agentic Applications 2026** (ASI01–ASI10) — applies to the agent + MCP server.
- **zod 4.4.3** — native `z.toJSONSchema()`; breaking vs v3: deprecated `.passthrough()`/`.strict()`, unified `error` param. Install `zod@^4`.
- **groq-sdk 1.2.1** — structured outputs `strict: true` (constrained decoding) limited to `openai/gpt-oss-20b`/`-120b`; no streaming or tool use with strict mode. Other models: best-effort `json_schema` or `json_object`.
- **@upstash/ratelimit 2.0.8** + `@upstash/redis` — current; sliding window + `Duration` strings, `analytics: true` optional.
- **promptfoo 0.121.x** — `redteam` command with `prompt-injection`/`indirect-prompt-injection` plugins and retry strategy (auto-incorporates past failures); runs against custom JS/HTTP providers; CI-friendly (`npx promptfoo eval`).
- Model-level robustness (instruction hierarchy training) keeps improving ([OpenAI](https://openai.com/index/prompt-injections/)) but all vendors state injection is unsolved — keep deterministic layers regardless of model upgrades.
