# Inngest for Incremental Data Pipelines — Best Practices
> Researched 2026-06-10. Sources verified against current official docs.

## TL;DR
- Use `inngest@4` (v4 GA 2026-03-16). Triggers now live in the options object; cloud mode is the default — set `INNGEST_DEV=1` locally.
- One `step.run` per side effect. Each step retries independently: 4 retries by default, exponential backoff with jitter.
- Idempotency at three layers: deterministic event `id` (24 h dedup), function-level `idempotency` CEL key, and DB upserts keyed on the HN item id (the only durable layer).
- Fan out per posting with `step.sendEvent` (one run per item) — never `step.run` in a loop over hundreds of comments (1,000-step cap).
- Cron + event trigger on the same function (`triggers: [...]`) so scheduled ingestion and manual backfills share one code path.
- Dead letters: `onFailure` per function plus a global `inngest/function.failed` listener that writes to a `failed_ingestions` table; use dashboard Replay for bulk recovery.

## Practices
- **v4 wiring in App Router.** Client in `lib/inngest/client.ts`: `new Inngest({ id: "hiring-radar", checkpointing: { maxRuntime: "50s" } })`. Route at `app/api/inngest/route.ts`: `export const { GET, POST, PUT } = serve({ client, functions })` from `inngest/next`, with `export const maxDuration = 60` so checkpointed multi-step requests don't get killed. https://www.inngest.com/docs/getting-started/nextjs-quick-start
- **Design steps as retryable transactions.** Each `step.run("name", fn)` is memoised after success and retried independently on failure — its return value must be JSON-serialisable (< 4 MB; run state cap 32 MB). Split "fetch thread", "diff against DB", "send fan-out events" into separate steps so a Groq outage never re-fetches HN. https://www.inngest.com/docs/learn/inngest-functions
- **Retries: default 4, tune per function.** Set `retries` in function options. Throw `NonRetriableError` when Groq output fails schema validation after repair attempts (retrying won't help); throw `RetryAfterError("rate limited", retryAfter)` on Groq/HN 429s to respect `Retry-After` instead of blind backoff. https://www.inngest.com/docs/features/inngest-functions/error-retries/retries
- **Event idempotency for the monthly ingest.** Send events with deterministic ids: `{ id: \`hn-post-${commentId}\`, name: "hn/posting.discovered", data: {...} }`. Dedup window is only 24 h, so for re-ingestion across days rely on Postgres `ON CONFLICT (hn_id) DO UPDATE` in the write step. Add `idempotency: "event.data.hnId"` on the per-posting function as a consumer-side guard. https://www.inngest.com/docs/guides/handling-idempotency
- **Fan out per posting, not batch steps.** The thread-level function diffs new/edited comments, then one `step.sendEvent("fan-out-postings", events)` (max 5,000 events/send, 256 KB/event payload — send the HN id, not the raw text, if comments run large). Each `hn/posting.discovered` run does parse → embed → upsert. You get per-item retries, per-item traces, and per-item Replay. https://www.inngest.com/docs/guides/fan-out-jobs https://www.inngest.com/docs/usage-limits/inngest
- **Batching only where writes dominate.** `batchEvents: { maxSize: 50, timeout: "30s" }` makes the handler receive `events[]` — useful if you later split embedding upserts into bulk pgvector writes. Note: batching is incompatible with `idempotency`, rate limiting, and priority. https://www.inngest.com/docs/guides/batching
- **Cron with explicit TZ, plus a manual trigger.** "Who is hiring" threads appear on the 1st ~11:00 ET and accrue comments for days. Use `triggers: [{ cron: "TZ=America/New_York 0 13 1-7 * *" }, { event: "hn/ingest.requested" }]` — daily re-polls during the first week pick up the delta; the event trigger reuses the same function for backfills. Up to 10 triggers per function; prefer `TZ=UTC` for DST-sensitive schedules. https://www.inngest.com/docs/guides/scheduled-functions https://www.inngest.com/docs/guides/multiple-triggers
- **Backfill = same pipeline, parameterised event.** Script sends `hn/ingest.requested` with `{ threadId, month }` per historical thread — DB upserts make it safe to re-run. For "we shipped a parser fix" recovery, use dashboard Replay: filter by function, time range, and status (Failed, or Succeeded for silent bugs). https://www.inngest.com/docs/platform/replay
- **Local dev vs deployed.** Locally: `npx inngest-cli@latest dev` (UI at `localhost:8288`, auto-discovers `/api/inngest` on port 3000) and `INNGEST_DEV=1` in `.env.local`; keys are not validated. Deployed: set `INNGEST_EVENT_KEY` + `INNGEST_SIGNING_KEY`, and sync the app in the Inngest dashboard after each deploy (Vercel integration does this automatically). https://www.inngest.com/docs/local-development
- **Observability comes free — feed it well.** Dashboard gives per-function metrics, step-level traces, event logs, and SQL-style Insights with zero instrumentation. Pass your Pino logger via `new Inngest({ logger })` (v4 uses Pino-style object-first structured logging) so app logs appear inside run traces. https://www.inngest.com/docs/platform/monitor/observability-metrics
- **Dead letters explicitly.** `onFailure` on the per-posting function runs once retries are exhausted — write the raw comment + error to a `failed_ingestions` table so the radar shows partial data instead of dropping postings. Add one global function triggered by `inngest/function.failed` for Sentry/alerting across the app. https://www.inngest.com/docs/features/inngest-functions/error-retries/failure-handlers

## Pitfalls
- **v3 tutorials everywhere.** The 3-arg `createFunction({ id }, { event }, handler)` signature is v3; v4 takes `{ id, triggers: { event } }` + handler. `EventSchemas().fromRecord<...>()` is removed — use `eventType("hn/posting.discovered", { schema: z.object({...}) })` for runtime-validated payloads. https://www.inngest.com/docs/reference/typescript/v4/migrations/v3-to-v4
- **v4 defaults to cloud mode** (v3 defaulted to dev). Without `INNGEST_DEV=1` the local app demands a signing key and signature verification fails silently-looking.
- **`step.run` in a loop** over thread comments hits the 1,000-step cap and bloats run state. Fan out or process the loop inside a single step.
- **Retries multiply per step.** `retries: 4` on a 3-step function allows up to 15 executions — every step must be idempotent (upserts, not inserts).
- **24 h idempotency window is shorter than your pipeline.** Daily re-polls of the same thread will re-emit events for edited comments after 24 h by design; the DB upsert is the source of truth, not the event id.
- **Batching silently disables idempotency config** — don't combine `batchEvents` with `idempotency` on the same function.
- **Checkpointing default change:** v4 runs multiple steps per HTTP request; on Vercel, an unset `checkpointing.maxRuntime` above the route's `maxDuration` causes mid-step timeouts. Set `maxRuntime` ~10 s below `maxDuration`.
- **`onFailure` is not a queue.** It fires once after final retry and can itself fail; persist to Postgres inside it rather than only alerting.
- **Renames:** `serveHost` → `serveOrigin` (`INNGEST_SERVE_ORIGIN`), `streaming: "force"/"allow"` → `true | false`, `logLevel` → `logger: new ConsoleLogger({ level })`.

## Version notes
Current as of June 2026: **`inngest@4.5.1`** (v4 GA 2026-03-16; v3 LTS line is `3.54.0` — don't start new work on it) and **`inngest-cli@1.27.0`** for the dev server. Install: `npm install inngest@latest`; run locally with `npx --ignore-scripts=false inngest-cli@latest dev`. v3→v4 breaking changes that matter here: triggers moved into function options; `signingKey`/`baseUrl` moved from `serve()` to the `Inngest` client; middleware rewritten as class-based (`Middleware.BaseMiddleware`); `step.invoke()` requires `referenceFunction()` or a function instance; checkpointing and optimised parallelism on by default; cloud mode default (set `INNGEST_DEV=1` locally). Migration guide: https://www.inngest.com/docs/reference/typescript/v4/migrations/v3-to-v4 — announcement: https://www.inngest.com/blog/typescript-sdk-v4.0
